# 智能车辆档案系统 — 项目分工文档

## 一、项目概述

**题目**：题目19 车型图像识别系统（扩展为智能车辆档案系统）

**核心思路**：三位成员每人训练一个视觉模型 + 开发配套工具，通过 LLM（大模型）的 Function Calling 能力调度三个模型协同工作，最终实现：**拍一张车 → 自动生成完整车辆档案**。

**输出档案示例**：
```
车型：奔驰 E300L 2023款 | 排量：2.0T / 258马力
车牌：京A·12345 | 归属：北京 | 类型：蓝牌小型车
车况：前保险杠轻微划痕 | 严重程度：轻度
预估维修：¥800-1200 | 二手车估价：¥38-42万
```

---

## 二、系统架构

```
用户拍照/上传（Web 前端）
    │
    ▼
┌─────────────────────────────────┐
│       LLM 智能调度中心            │
│   (DeepSeek API Function Call)   │
│                                  │
│   接收图片 → 决定调用哪些模型      │
│   拿到结果 → 整合分析 → 生成档案   │
└──┬──────────┬──────────┬────────┘
   │          │          │
   ▼          ▼          ▼
成员A      成员B      成员C
车型识别   车牌识别   车况检测
+ 工具     + 工具     + 工具
```

## 三、三人分工

### 🚗 成员A：车型识别 + 车辆信息工具

**模型训练**
- 任务：给定车辆图片，识别品牌 + 具体型号（如"奔驰 E300L 2023款"）
- 推荐架构：EfficientNet-B3 / ResNet50 / ViT 微调
- 数据集：Stanford Cars（196类）、CompCars

**开发的工具**

| 工具名 | 功能 | 数据来源 |
|---|---|---|
| `query_vehicle_params` | 根据品牌+型号+年份查询详细参数（排量、马力、油耗、配置等） | 汽车之家/懂车帝爬虫 或 开源车型参数库 |
| `estimate_market_price` | 根据车型+年份+车况估算二手车价格区间 | 瓜子/人人车等平台价格参考 |

**对外 API**（FastAPI）
```
POST /api/vehicle/infer          ← 车型识别推理
POST /api/vehicle/tools/params   ← 查询车辆参数
POST /api/vehicle/tools/price    ← 估价
```

---

### 🔢 成员B：车牌检测识别 + 车主关联工具

**模型训练**
- 任务：从图片中检测车牌位置 → 识别车牌号（中文+字母+数字）
- 推荐架构：YOLOv8（车牌检测）+ PaddleOCR（字符识别）
- 数据集：CCPD（中文车牌数据集）

**开发的工具**

| 工具名 | 功能 | 数据来源 |
|---|---|---|
| `query_plate_info` | 根据车牌号返回归属地、车辆类型（蓝牌/绿牌等） | 车牌编码规则解析 |
| `check_violation` | 查询违章记录（可模拟数据或对接第三方API） | 模拟数据 / 交管12123 API |
| `query_vehicle_history` | 查询年检状态、维保记录概要 | 模拟数据 |

**对外 API**（FastAPI）
```
POST /api/plate/infer               ← 车牌检测+识别推理
POST /api/plate/tools/plate_info    ← 车牌归属信息
POST /api/plate/tools/violation     ← 违章查询
POST /api/plate/tools/history       ← 年检/维保记录
```

---

### 🔧 成员C：车况检测 + 维修方案工具

**模型训练**
- 任务：检测车辆损伤类型（划痕、凹陷、玻璃碎裂、漆面脱落等），判断严重程度
- 推荐架构：EfficientNet 多标签分类 / YOLOv8 分割
- 数据集：Car Damage Detection (Kaggle)

**开发的工具**

| 工具名 | 功能 | 数据来源 |
|---|---|---|
| `diagnose_damage` | 根据损伤类型+严重程度给出诊断分析 | 维修知识库整理 |
| `estimate_repair` | 根据诊断结果生成维修方案 + 预估费用 | 维修工时+配件价格参考 |
| `recommend_insurance` | 判断是否建议走保险理赔，给出建议 | 保险理赔规则整理 |

**对外 API**（FastAPI）
```
POST /api/damage/infer               ← 车况检测推理
POST /api/damage/tools/diagnose      ← 损伤诊断
POST /api/damage/tools/repair        ← 维修方案+费用
POST /api/damage/tools/insurance     ← 保险建议
```

## 四、API 统一契约（⚠️ 重要，必须遵守）

三个人暴露的 API 格式必须一致，否则 LLM 调度层接不上。

### 推理接口
```json
// POST /api/{模块}/infer
// 请求
{
    "image": "base64编码的图片字符串"
}

// 响应
{
    "status": "ok",
    "result": {
        // 成员A示例：{ "brand": "奔驰", "model": "E300L", "year": "2023", "confidence": 0.94 }
        // 成员B示例：{ "plate": "京A12345", "plate_type": "蓝牌", "location": "北京", "confidence": 0.97 }
        // 成员C示例：{ "conditions": ["scratch_front"], "severity": "mild", "confidence": 0.89 }
    },
    "latency_ms": 120
}
```

### 工具接口
```json
// POST /api/{模块}/tools/{工具名}
// 请求
{
    "params": {
        // 由各工具自己定义所需参数
    }
}

// 响应
{
    "status": "ok",
    "data": {
        // 工具返回的具体数据
    }
}
```

### 健康检查
```json
// GET /api/{模块}/health
{
    "status": "ok",
    "model_loaded": true,
    "model_name": "efficientnet-b3-cars"
}
```

## 五、LLM Tool Definition（每人需提供）

每个工具的 JSON Schema 定义，用于注册到 DeepSeek Function Calling。

**格式示例**（以成员A的参数查询工具为例）：

```json
{
    "type": "function",
    "function": {
        "name": "query_vehicle_params",
        "description": "根据车型品牌、型号和年份查询车辆详细参数，包括排量、马力、油耗、配置等",
        "parameters": {
            "type": "object",
            "properties": {
                "brand": { "type": "string", "description": "车辆品牌，如'奔驰'" },
                "model": { "type": "string", "description": "具体型号，如'E300L'" },
                "year":  { "type": "string", "description": "年份，如'2023'" }
            },
            "required": ["brand", "model", "year"]
        }
    }
}
```

每人需要为自己的所有工具提供这样的 schema 定义，统一放在 `TOOLS_SCHEMA.md` 中。

## 六、技术栈

| 层 | 技术 | 说明 |
|---|---|---|
| 视觉模型 | Python + PyTorch / TensorFlow | 三人各自训练 |
| 模型服务 | FastAPI | 三人各自封装为 API |
| LLM 调度 | Node.js + DeepSeek API | 负责 Function Calling + 结果整合 |
| 前端 | 原生 HTML/CSS/JS | 图片上传 + 档案展示 + 对话 |
| 存储 | SQLite（轻量）或 PostgreSQL | 存档车辆档案 |
| 部署 | Docker 容器化 | 每个服务一个容器 |

## 八、协作约定

1. **API 契约优先**：先定接口格式，再写实现。改了接口要及时同步。
2. **每人独立 namespace**：URL 路径用 `/api/vehicle`、`/api/plate`、`/api/damage` 区分，互不冲突。
3. **Git 分支管理**：每人一个分支（`member-a`、`member-b`、`member-c`），稳定后合并到 `main`。
4. **及时沟通**：遇到数据集/模型问题尽早群里说，别自己闷头搞。
